## Description

Update the existing GitHub Actions workflow (`.github/workflows/deploy.yml`) to run ESLint and Vitest before the build step. The pipeline currently only installs dependencies and builds -- there are no quality gates. If lint or tests fail, the pipeline should fail and the deployment should not proceed.

This is a small but critical story: it ensures that code quality checks are enforced on every push to `main`, preventing regressions from being deployed.

---

## Acceptance Criteria

- [ ] `npm run lint` step runs after dependency install and before build
- [ ] `npm test` step runs after dependency install and before build
- [ ] Pipeline fails immediately if lint errors are found (non-zero exit code)
- [ ] Pipeline fails immediately if any test fails (non-zero exit code)
- [ ] Build step only executes if both lint and test steps pass
- [ ] Pipeline still deploys to GitHub Pages on success (no regression to existing deploy behavior)
- [ ] Workflow uses `npm ci` for dependency installation (already in place)
- [ ] Node version matches project requirements (Node 20)

## Technical Context

### Current State
- **`.github/workflows/deploy.yml`** exists with a `build` job that runs: Checkout -> Setup Node 20 -> Setup Pages -> `npm ci` -> `npm run build` -> Upload artifact
- A separate `deploy` job deploys the artifact to GitHub Pages
- **`npm run lint`** is configured in `package.json` as `next lint` (ESLint with eslint-config-next)
- **`npm test`** is configured in `package.json` as `vitest run` (single run, exits with code)
- Both scripts are already functional locally; they just need to be added to the CI workflow

### Architecture Decisions
- **ADR: Steps within the same job, not separate jobs** -- Lint and test run in the same `build` job to avoid re-installing dependencies in parallel jobs. The total runtime is short enough (~2-3 minutes) that parallelism is unnecessary, and sequential steps within one job share the node_modules cache.
- **ADR: Lint before test before build** -- Lint is fastest and catches the most trivial issues first. Tests run next. Build runs last (most expensive). This fail-fast ordering minimizes wasted CI minutes.

### Relevant Files
- `/.github/workflows/deploy.yml` -- the only file to modify
- `/package.json` -- contains the `lint` and `test` scripts (no changes needed)

## Implementation Tasks

### Task 1: Add lint step to deploy.yml (est. 5 min)

Insert a new step after "Install dependencies" and before "Build":

```yaml
      - name: Lint
        run: npm run lint
```

`next lint` will exit with a non-zero code if there are any ESLint errors, which causes the step (and therefore the job) to fail.

### Task 2: Add test step to deploy.yml (est. 5 min)

Insert a new step after "Lint" and before "Build":

```yaml
      - name: Test
        run: npm test
```

`vitest run` will exit with a non-zero code if any tests fail, which causes the step (and therefore the job) to fail.

### Task 3: Verify locally (est. 10 min)

Run both commands locally to confirm they pass before pushing:
- `npm run lint` -- should exit 0 with no errors
- `npm test` -- should exit 0 with all tests passing
- `npm run build` -- should exit 0 producing the static export

## Notes

- **The change is minimal** -- only two new YAML steps in the existing workflow file. The risk of regression is very low.
- **If lint has warnings but no errors**, the pipeline will still pass. `next lint` only fails on errors by default. This is acceptable for now; a future story could add `--max-warnings 0` to also fail on warnings.
- **Test failures will block deployment**, which is the desired behavior. If a flaky test is introduced later, it should be fixed rather than bypassed.
- **No changes to `package.json` are needed** -- the `lint` and `test` scripts already exist from S-001 and the project scaffolding.
