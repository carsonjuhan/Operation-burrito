## Description

Write component/integration tests for the most critical user flows: Dashboard rendering, Items page CRUD, Birth Plan auto-save, Contraction Timer start/stop with 5-1-1 rule detection, and Settings sync UI states. These tests exercise the full React component tree with a mock `StoreContext` provider, ensuring the UI renders correctly and user interactions trigger the expected state changes.

The testing framework (Vitest + React Testing Library) was set up in S-001. Unit tests for utility modules were written in S-002. This story adds component-level tests that render actual page components with mocked store data.

---

## Acceptance Criteria

- [ ] Dashboard test: renders stats (items purchased, bag packed, birth plan fields, classes), countdown banner, quick links, budget strip, must-have items list, upcoming appointments, and pinned notes with mock store data
- [ ] Items page test: renders items list, can add a custom item via modal form, can mark an item as purchased, can delete an item
- [ ] Birth plan test: renders birth plan form, fields trigger `updateBirthPlan` on change (auto-save)
- [ ] Contraction timer test: Start button begins timing, Stop button records a contraction, 5-1-1 rule triggers alert banner when criteria are met
- [ ] Settings test: renders connect form when disconnected, renders connected state with push/pull buttons when connected
- [ ] All tests use a mock `StoreContext` provider with configurable initial data
- [ ] All tests pass via `npm test`
- [ ] `npm run build` still succeeds

## Technical Context

### Current State
- Vitest is configured with `jsdom` environment, `globals: true`, and `@/*` path alias (from S-001)
- Test setup file at `src/test/setup.ts` imports `@testing-library/jest-dom/vitest`
- `StoreContext` is in `src/contexts/StoreContext.tsx`, exposes `useStoreContext()`
- All pages use `"use client"` and `useStoreContext()` to access store data and actions
- Pages gate on `loaded` flag before rendering content

### Architecture Decisions
- **Mock StoreContext provider** -- create a `MockStoreProvider` that wraps components with pre-configured store data and mock action functions
- **Mock `next/link`** and `next/navigation` -- since we test in jsdom without Next.js router
- **Mock `@/lib/checklistData`** -- the Items page imports heavy checklist data; mock it to avoid loading real data files
- **Mock `@/lib/gistSync`** -- the Settings page imports gist sync functions; mock them
- **Use `@testing-library/user-event`** for realistic user interactions (click, type, etc.)

### Relevant Source Files
- `src/app/page.tsx` -- Dashboard page
- `src/app/items/page.tsx` -- Items page with CRUD
- `src/app/birth-plan/page.tsx` -- Birth plan builder
- `src/app/timer/page.tsx` -- Contraction timer
- `src/app/settings/page.tsx` -- Settings/sync page
- `src/contexts/StoreContext.tsx` -- Store context provider
- `src/hooks/useStore.ts` -- Store hook with all actions
- `src/types/index.ts` -- TypeScript interfaces

## Implementation Tasks

### Task 1: Create mock StoreContext provider (~20 min)

Create `src/test/mockStoreProvider.tsx`:
- Export a `MockStoreProvider` component that wraps children with `StoreContext.Provider`
- Accept partial store overrides and mock action functions
- Default `loaded: true` so pages render content immediately
- All action functions default to `vi.fn()` (no-op)
- Export factory helpers for creating mock store data (items, appointments, notes, etc.)

### Task 2: Write Dashboard rendering tests (~30 min)

Create `src/app/__tests__/Dashboard.test.tsx`:
- Renders "Dashboard" heading
- Shows due date countdown banner when birth plan has a due date
- Shows stat cards with correct values (items purchased, bag packed, birth plan fields, classes)
- Shows budget strip when items have estimated costs
- Shows must-have items that are not purchased
- Shows upcoming appointments sorted by date
- Shows pinned notes
- Handles empty store gracefully (no crash, shows empty states)

### Task 3: Write Items page CRUD tests (~40 min)

Create `src/app/__tests__/Items.test.tsx`:
- Mock `@/lib/checklistData` to provide minimal test data
- Renders items page header
- Opens add modal on "Add Custom" button click
- Submitting form calls `addItem` with correct payload
- Calls `deleteItem` when delete button is clicked
- Calls `updateItem` to toggle purchased status

### Task 4: Write Birth Plan auto-save tests (~25 min)

Create `src/app/__tests__/BirthPlan.test.tsx`:
- Renders birth plan form with personal info tab
- Typing in Legal Name field triggers `updateBirthPlan`
- Changing a checkbox triggers `updateBirthPlan`
- Renders field values from mock store data

### Task 5: Write Contraction Timer tests (~30 min)

Create `src/app/__tests__/Timer.test.tsx`:
- Renders "Contraction Timer" heading and Start button in idle state
- Clicking Start shows Stop button and "Contraction in progress" label
- Clicking Stop calls `addContraction` and shows "Between contractions" label
- Shows 5-1-1 rule alert banner when provided contractions meet criteria
- Shows contraction history table when contractions exist

### Task 6: Write Settings sync UI tests (~25 min)

Create `src/app/__tests__/Settings.test.tsx`:
- Mock `@/lib/gistSync` module
- Renders connect form when no PAT is saved (disconnected state)
- Shows "GitHub Connected" when PAT is saved
- Shows push/pull buttons when connected

### Task 7: Verify all tests pass (~10 min)

- Run `npm test` and confirm all tests pass
- Run `npm run build` to confirm no build breakage

## Notes

- **`next/link`** renders as `<a>` in jsdom which is fine for testing text and attributes
- **`next/navigation`** may need mocking if any component uses `useRouter()` or `usePathname()`
- **`crypto.randomUUID()`** is used by store actions; mock it for deterministic test IDs
- **Timer tests** should use `vi.useFakeTimers()` to control `setInterval` and `Date.now()`
- **Items page** imports from `@/lib/checklistData` which contains large data arrays; mock the entire module to keep tests fast and focused
- **Settings page** imports from `@/lib/gistSync`; mock all exported functions
- The `loaded` flag in the mock provider must be `true` for pages to render content (they return `null` or a loading indicator when `loaded` is false)
