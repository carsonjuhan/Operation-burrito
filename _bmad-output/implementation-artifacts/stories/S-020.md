## Description

Add a visual indicator when the browser goes offline and implement a robust sync queue that persists pending operations in localStorage. When connectivity returns, queued sync operations execute automatically. This builds on the offline queue infrastructure created in S-018 (`useServiceWorker.ts`) by adding user-facing UI and automatic replay logic.

---

## Acceptance Criteria

- [ ] A banner or badge appears when the browser goes offline
- [ ] Banner disappears when connectivity is restored
- [ ] Sync operations attempted while offline are queued in localStorage
- [ ] Queued syncs execute automatically when online status returns
- [ ] Queue is persisted in localStorage to survive page refreshes
- [ ] Toast notification when coming back online with queued operations
- [ ] Unit tests cover the offline detection hook and sync queue logic

---

## Technical Context

### Current State
- S-018 created `src/hooks/useServiceWorker.ts` with `queueOfflineSync()`, `getOfflineSyncQueue()`, and `clearOfflineSyncQueue()` utility functions
- The service worker in `public/sw.js` posts `SYNC_OFFLINE` messages when Gist API calls fail due to network
- `useStore.ts` handles auto-sync with debounced `pushToGist` calls
- `ToastContext.tsx` provides the toast notification system
- `Sidebar.tsx` shows sync status in the footer area
- The layout renders `<ServiceWorkerRegistration />` for SW registration

### Architecture Decisions
- **ADR: Hook-based offline detection** -- A `useOnlineStatus` hook wraps the browser `online`/`offline` events and `navigator.onLine` for reactive status. This is a pure client-side approach with no server dependency.
- **ADR: Banner in layout, not sidebar** -- The offline banner is rendered at the top of the main content area (inside `layout.tsx`) so it is visible regardless of sidebar state on mobile.
- **ADR: Sync queue replay via useStore integration** -- When coming back online, the hook triggers `retrySyncNow()` from the store context if there are queued operations, reusing the existing sync infrastructure.

### Relevant Files
- `src/hooks/useServiceWorker.ts` -- existing offline queue helpers
- `src/hooks/useStore.ts` -- auto-sync and retry logic
- `src/components/Sidebar.tsx` -- sync status display
- `src/contexts/ToastContext.tsx` -- toast notifications
- `src/app/layout.tsx` -- root layout

## Implementation Tasks

### Task 1: Create useOnlineStatus hook (est. 15 min)

Create `src/hooks/useOnlineStatus.ts`:
- Track `navigator.onLine` state reactively
- Listen to `online` and `offline` window events
- Return `{ isOnline: boolean }` for consumers
- Clean up event listeners on unmount

### Task 2: Create OfflineBanner component (est. 15 min)

Create `src/components/OfflineBanner.tsx`:
- `"use client"` component using `useOnlineStatus`
- Shows a fixed-position banner at the top when offline
- Banner has WifiOff icon, message text, and appropriate styling
- Accessible with role="alert" for screen readers
- Animates in/out smoothly

### Task 3: Create SyncQueueManager component (est. 20 min)

Create `src/components/SyncQueueManager.tsx`:
- `"use client"` component that monitors online status changes
- When transitioning from offline to online, checks for queued operations
- Triggers sync replay via `retrySyncNow()` from store context
- Shows toast notification about replaying queued operations
- Clears the queue after successful replay

### Task 4: Integrate into layout (est. 10 min)

- Add `OfflineBanner` to `layout.tsx` above main content
- Add `SyncQueueManager` to `layout.tsx` for background sync replay

### Task 5: Add offline badge to Sidebar (est. 10 min)

- Show a small offline badge/indicator in the Sidebar sync status area
- Override normal sync status text when offline

### Task 6: Write tests (est. 30 min)

- Test `useOnlineStatus` hook with mocked browser events
- Test `OfflineBanner` renders when offline, hides when online
- Test `SyncQueueManager` triggers sync replay on reconnection
