## Description

Install and configure Vitest with React Testing Library as the project's testing foundation. The codebase currently has zero test files and no test framework configured. This story establishes the testing infrastructure that every subsequent test story (S-002, S-003, S-004) depends on.

Vitest is chosen over Jest because:
- Native ESM support (aligns with Next.js and the `"module": "esnext"` tsconfig)
- Built-in TypeScript support without additional transform config
- Faster execution via Vite's dependency pre-bundling
- Compatible `expect`/`describe`/`it` API (easy migration path if needed)

## Technical Context

### Current State
- **package.json** has no test script and no test dependencies
- **tsconfig.json** uses `"module": "esnext"`, `"moduleResolution": "bundler"`, strict mode, and `@/*` path alias mapping to `./src/*`
- **No test files** exist anywhere in the codebase
- The project uses **Next.js 14.2.5** with App Router and `output: "export"` (static site)
- All pages use `"use client"` -- components are standard React client components

### Architecture Decisions
- **ADR: Vitest over Jest** -- Vitest provides zero-config TypeScript/ESM support that aligns with the project's bundler module resolution. Jest would require additional transform configuration (`ts-jest` or `@swc/jest`).
- **ADR: jsdom over happy-dom** -- jsdom is the more mature and widely-used DOM implementation. happy-dom is faster but has more edge cases with complex DOM APIs.

### Relevant Files
- `/package.json` -- add dependencies and scripts
- `/tsconfig.json` -- verify test file inclusion (already uses `**/*.ts` glob)
- `/src/contexts/StoreContext.tsx` -- the store context the smoke test will exercise
- `/src/app/layout.tsx` -- root layout showing component hierarchy
- `/src/components/EmptyState.tsx` -- simple component, good candidate for smoke test

## Implementation Tasks

### Task 1: Install dependencies (est. 10 min)

```bash
npm install -D vitest @vitejs/plugin-react jsdom @testing-library/react @testing-library/jest-dom @testing-library/user-event
```

Packages:
- `vitest` -- test runner
- `@vitejs/plugin-react` -- React JSX transform for Vitest
- `jsdom` -- DOM environment for component tests
- `@testing-library/react` -- React component testing utilities
- `@testing-library/jest-dom` -- custom matchers (toBeInTheDocument, toHaveTextContent, etc.)
- `@testing-library/user-event` -- user interaction simulation (needed for S-003, install now to avoid churn)

### Task 2: Create Vitest configuration (est. 15 min)

Create `vitest.config.ts` at project root:

```typescript
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/test/setup.ts'],
    include: ['src/**/*.test.{ts,tsx}'],
    css: false,
  },
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },
});
```

Key decisions:
- `globals: true` -- enables `describe`, `it`, `expect` without imports (matches Jest convention)
- `css: false` -- skip CSS processing in tests (Tailwind classes are just strings)
- `setupFiles` -- runs before each test file for global setup (jest-dom matchers)

### Task 3: Create test setup file (est. 5 min)

Create `src/test/setup.ts`:

```typescript
import '@testing-library/jest-dom/vitest';
```

This registers custom matchers like `toBeInTheDocument()`, `toHaveTextContent()`, etc. globally for all test files.

### Task 4: Add TypeScript types for test globals (est. 5 min)

Create `src/test/vitest.d.ts` (or add to tsconfig):

```typescript
/// <reference types="vitest/globals" />
/// <reference types="@testing-library/jest-dom/vitest" />
```

Ensure `tsconfig.json` includes these type references so IDE autocomplete works.

### Task 5: Add test scripts to package.json (est. 5 min)

Add to `scripts`:
```json
{
  "test": "vitest run",
  "test:watch": "vitest",
  "test:coverage": "vitest run --coverage"
}
```

- `test` -- single run for CI (exits with code)
- `test:watch` -- interactive watch mode for development
- `test:coverage` -- coverage report (useful for future stories)

### Task 6: Write smoke test (est. 20 min)

Create `src/components/__tests__/EmptyState.test.tsx`:

```typescript
import { render, screen } from '@testing-library/react';
import { EmptyState } from '../EmptyState';

describe('EmptyState', () => {
  it('renders the message and action button', () => {
    render(
      <EmptyState
        icon={...}
        title="No items yet"
        message="Add your first item to get started"
        actionLabel="Add Item"
        onAction={() => {}}
      />
    );
    expect(screen.getByText('No items yet')).toBeInTheDocument();
    expect(screen.getByText('Add Item')).toBeInTheDocument();
  });
});
```

The smoke test should:
1. Import and render a real component from the codebase
2. Assert on rendered text content
3. Prove that path aliases, JSX transform, jsdom, and Testing Library all work together

**Note:** Review the actual `EmptyState` component props before writing the test. The example above is illustrative -- adapt to the real API.

### Task 7: Verify everything works together (est. 10 min)

Run each command and verify:
- `npm test` -- smoke test passes, exits 0
- `npm run build` -- build still succeeds
- `npm run lint` -- lint still passes
- TypeScript: no new type errors in IDE

## Notes

- **Do NOT install Jest.** Vitest is the chosen runner. If you see references to Jest config files or jest.config.js, that is incorrect for this project.
- **The `@vitejs/plugin-react` package is required** for JSX transform in Vitest. Without it, `.tsx` test files will fail to parse.
- **CSS processing is disabled** in tests (`css: false`). Tailwind class names are just strings in the DOM -- no need to process them. This significantly speeds up test execution.
- **The `globals: true` setting** means you do NOT need `import { describe, it, expect } from 'vitest'` at the top of every test file. They are available globally, matching the Jest convention most developers expect.
- **Watch mode** (`npm run test:watch`) is the primary development workflow. `npm test` (single run) is for CI.
- **EmptyState was chosen for the smoke test** because it is a simple presentational component with no store dependency. This isolates the test framework verification from any state management complexity.
- **`@testing-library/user-event`** is installed now even though it is not needed until S-003. Installing it in this story avoids a separate dependency-install step later and keeps the test toolchain complete from the start.
- If the `EmptyState` component has a different prop interface than assumed above, adapt the smoke test accordingly. The goal is to render something and assert -- the specific component is secondary.
