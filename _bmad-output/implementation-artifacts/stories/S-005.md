## Description

Add React Error Boundary components to catch runtime errors and display a user-friendly fallback instead of a blank white screen. Each page route should be wrapped independently so that a crash in one page does not take down the sidebar, navigation, or other pages.

React error boundaries must be class components (the `componentDidCatch` / `getDerivedStateFromError` lifecycle methods are not available to function components). We will create a reusable `ErrorBoundary` class component and a `PageErrorBoundary` wrapper that provides styled fallback UI consistent with the app's design system.

The approach uses Next.js App Router `error.tsx` convention files for each route segment, plus a reusable `ErrorBoundary` component for any ad-hoc usage within pages.

## Acceptance Criteria

- [ ] A reusable `ErrorBoundary` class component is created in `src/components/ErrorBoundary.tsx`
- [ ] A `PageErrorBoundary` wrapper component provides styled fallback UI with a friendly message and "Reload" button
- [ ] Next.js `error.tsx` files are created for each page route directory under `src/app/`
- [ ] A root-level `src/app/error.tsx` catches errors from the dashboard page
- [ ] A root-level `src/app/global-error.tsx` catches errors from the root layout itself
- [ ] Sidebar and navigation remain functional when a page crashes (only the `<main>` content shows the error fallback)
- [ ] Error details are logged to `console.error` for debugging
- [ ] Fallback UI uses the project's design system (sage/blush/stone palettes, card styling)
- [ ] The "Reload" button calls `reset()` to attempt re-rendering, with a "Reload Page" fallback that does a full page reload
- [ ] Unit tests verify the ErrorBoundary catches errors and renders fallback UI
- [ ] `npm run build` and `npx vitest run` pass with no regressions

## Technical Context

### Current State
- `src/app/layout.tsx` renders `AuthGate` > `StoreProvider` > `Sidebar` + `<main>{children}</main>`
- No error boundaries exist anywhere in the app
- No `error.tsx` or `global-error.tsx` files exist
- All pages use `"use client"` and consume `useStoreContext()`

### Implementation Plan

1. **`src/components/ErrorBoundary.tsx`** -- Reusable class component
   - Props: `children`, `fallback` (optional custom fallback), `onError` callback
   - State: `hasError`, `error`
   - `getDerivedStateFromError` sets `hasError = true`
   - `componentDidCatch` logs error to console
   - Default fallback renders a styled card with error message and retry button

2. **`src/components/PageErrorFallback.tsx`** -- Styled fallback component
   - Shows an `AlertTriangle` icon, friendly heading, error message (in dev), and two buttons: "Try Again" (reset) and "Reload Page" (window.location.reload)
   - Matches the app's `card` styling with sage/stone colors

3. **Next.js `error.tsx` files** -- One per route directory
   - Each exports a `"use client"` component that receives `{ error, reset }` props
   - Renders `PageErrorFallback` with those props
   - Created for: root, appointments, birth-plan, budget, classes, contacts, hospital-bag, items, materials, notes, search, settings, timer

4. **`src/app/global-error.tsx`** -- Catches root layout errors
   - Must render its own `<html>` and `<body>` tags (Next.js requirement)
   - Minimal styling since globals.css may not load

5. **Tests** -- `src/components/__tests__/ErrorBoundary.test.tsx`
   - Renders children normally when no error
   - Catches thrown errors and shows fallback
   - Reset button clears error state
   - onError callback is invoked
