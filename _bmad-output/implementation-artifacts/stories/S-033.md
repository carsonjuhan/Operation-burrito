## Description

The current single React Context (`StoreContext`) causes all consuming components to re-render on any store mutation, regardless of which domain changed. For example, updating a note triggers re-renders in the Items page, Dashboard, Sidebar, and every other consumer.

This story splits the monolithic context into domain-specific contexts so that components only re-render when their specific data changes. The approach preserves the single localStorage key for persistence and maintains Gist auto-sync across all domains.

### Approach: Domain-Specific Contexts with Shared Persistence

Rather than migrating to Zustand (which would be a larger change with dependency additions), the implementation splits the single context into multiple focused contexts:

1. **ItemsContext** -- items CRUD, used by Items page, Budget page, Dashboard
2. **BirthPlanContext** -- birth plan updates, used by Birth Plan page, Dashboard
3. **NotesContext** -- notes CRUD, used by Notes page, Dashboard
4. **HospitalBagContext** -- bag items CRUD, used by Hospital Bag page, Dashboard
5. **AppointmentsContext** -- appointments CRUD, used by Appointments page, Dashboard
6. **ContactsContext** -- contacts CRUD, used by Contacts page
7. **ClassesContext** -- classes CRUD, used by Classes page, Dashboard
8. **MaterialsContext** -- materials CRUD, used by Materials page
9. **ContractionsContext** -- contractions, used by Timer page
10. **CoreContext** -- loaded state, autoSyncing, storageInfo, sync error management, registryUrl, loadFromExternal

A coordinating `StoreProvider` composes all contexts and owns the shared persistence/sync logic. Each domain context only triggers re-renders for its specific consumers.

## Technical Context

### Current State
- Single `StoreContext` in `src/contexts/StoreContext.tsx` exposes all store data and all CRUD actions
- `useStore` hook in `src/hooks/useStore.ts` manages all state, persistence, and auto-sync
- 26 files import `useStoreContext` -- every page and the Sidebar
- Tests mock `useStoreContext` via `vi.mock("@/contexts/StoreContext")`
- `mockStoreProvider.tsx` provides factory helpers for test data

### Architecture Decisions
- **ADR: Keep single localStorage key** -- Splitting localStorage keys per domain would complicate sync (Gist expects the full AppStore) and make import/export harder. The single key is preserved; the `update` function still saves the complete store.
- **ADR: Domain contexts over Zustand** -- Adding Zustand would require a new dependency and significantly change the testing patterns. Domain contexts achieve the same re-render optimization using only React primitives.
- **ADR: Backward-compatible useStoreContext** -- The existing `useStoreContext()` hook is preserved as a convenience that composes all domain hooks. This means existing code continues to work without changes, and pages can be incrementally migrated to use domain-specific hooks for better performance.

### Relevant Files
- `src/contexts/StoreContext.tsx` -- current monolithic context (to be refactored)
- `src/hooks/useStore.ts` -- current store hook with all state and CRUD (to be refactored)
- `src/test/mockStoreProvider.tsx` -- test mock provider (to be updated)
- All page files in `src/app/*/page.tsx` -- consumers (no changes required initially)
